# 欲望の小道の原理（Desire Path Principle）

## 一言で

**システムの「設計された道」と「実際に歩かれた道」のズレは、バグではなくシグナル。ズレを矯正するのではなく、舗装する。**

---

## 構造

都市計画の「desire path（欲望の小道）」から抽出した原理。

芝生に舗装された歩道があるのに、人は芝生を斜めに横切って近道する。
踏みならされた泥の道ができる。

設計者の反応は2つに分かれる：
1. ❌ 柵を立てて「正しい道」を歩かせる（= Frozen 化。ルールで矯正）
2. ✅ 泥の道を舗装する（= 実際の使い方に設計を合わせる）

### 一般化

```
計画された構造 ≠ 実際の使われ方
    ↓
このギャップ = 最も価値のある改善情報
    ↓
ギャップを「逸脱」と見るか「シグナル」と見るかで設計の質が分かれる
```

### 臨界点の原理との接続

- 計画しすぎ → Frozen（柵だらけで誰も歩けない）
- 計画なし → Disordered（道がなくて迷子になる）
- 欲望の小道を舗装 → **Coherent Phase**（構造はあるが、使い手が構造を育てられる）

### 生きた緊張の原理との接続

「計画 vs 実際」の緊張を**解消しない**（どちらかに統一しない）。
両方を生かしたまま住む = 計画は骨格、実際の使い方が肉付け。

## 使いどころ

| 場面 | 設計された道 | 欲望の小道 | アクション |
|------|------------|-----------|----------|
| **ルール設計** | 書かれたルール | 実際に発火してるルール | 使われないルールを削除、実際のパターンをルール化 |
| **タスク管理** | today.mdの計画 | 実際にやったこと | drift分析で検出、計画を実態に合わせる |
| **スキル** | .claude/commands/ | 実際に呼ばれるスキル | 使われないスキルを統合、よく使うパターンをスキル化 |
| **リサーチ→実装** | 知識が skills/ に貯まる | 知識が projects/ で使われる | **橋がなければ作る** |
| **UX設計** | ワイヤーフレーム上の導線 | ヒートマップの実際のクリック | 実態に合わせてUI再配置 |
| **チーム** | 組織図上の報告ライン | 実際の情報フロー | 実態のハブを公式化する |

## 診断方法（life-os用）

定期的に（/self-review で）以下を確認：

1. **git log のコミット分布** = 「どこを実際に歩いてるか」
2. **rules/ の各ルールが最後に発火した日** = 「使われてる道か」
3. **skills/ → projects/ への引用頻度** = 「知識の橋が通ってるか」
4. **curiosity.md の✅ vs 🔍の比率** = 「道が掃除されてるか」

## Alexander からの補強

Christopher Alexander の「生きたパターンランゲージ」の3条件：
1. **使う人がパターンを追加・修正できる**（life-os: お兄ちゃんもミナもルールを変えられる ✅）
2. **パターンには「力の緊張」が明示されている**（life-os: ルールに Why が書かれてない ❌ → 改善対象）
3. **適用順序（generative sequence）が存在する**（life-os: /morning → 作業 → /close-day の順序あり ✅）

Alexander の15の性質のうち、life-os で弱いもの：
- **Roughness**（雑さ）: ルールが整いすぎ。「だいたいこんな感じ」で良い部分まで精密に規定してる
- **The Void**（空白）: すべてルール化しようとしてる。「ルールがない余白」が創造性の場所なのに
- **Gradients**（グラデーション）: 全ルールが同じ強度。「絶対守る」と「緩くていい」の温度差がない

---

## 限界

- desire path は**過去の行動パターン**に基づく。未来の望ましい行動変化を導くには不十分
- 「歩かれてない道」が「不要な道」とは限らない。まだ発見されてないだけかもしれない
- 定量化が難しい。git log はプロキシだが完全な計測ではない

---

## 次に掘りたいこと

1. **Alexander の「generative sequence」を life-os に適用する** — ルールの適用順序を設計する
2. **ルールにgradient（温度）を導入する** — must / should / may の3段階
3. **void（空白）を意図的に設計する** — 「ルールがない領域」を明示する
